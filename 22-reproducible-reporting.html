22  Reproducible Workflows & Best Practices

Reproducibility is a non-negotiable standard for computational mass spectrometry. A published analysis must be rerunnable by others — and by your future self — months or years later. This chapter covers the tools and practices that transform ad-hoc scripts into robust, auditable, and shareable pipelines.

22.1 Why Reproducibility Matters in MS Analysis

  • MS data processing is highly parameter-sensitive: different peak-picking thresholds produce different feature lists
  • Bioconductor package updates can change results — dependency pinning is essential
  • Large datasets require pipeline frameworks to avoid manual reruns
  • Journals increasingly require code and data deposition alongside manuscripts

22.2 Package Management with renv

renv creates a project-local library that records the exact version of every package used, enabling exact reproduction of the computational environment.

22.2.1 What gets recorded

renv generates renv.lock — a JSON file listing all packages, their versions, and their source (CRAN, Bioconductor, GitHub). Commit this file to version control.

{
  "R": { "Version": "4.4.1" },
  "Packages": {
    "Spectra": {
      "Package": "Spectra",
      "Version": "1.14.0",
      "Source": "Bioconductor"
    },
    "xcms": {
      "Package": "xcms",
      "Version": "4.2.0",
      "Source": "Bioconductor"
    }
  }
}

22.3 Pipeline Orchestration with targets

targets is a Make-like pipeline framework for R. It tracks which steps are up to date and only reruns what has changed — critical for expensive MS preprocessing steps.

22.3.1 Core concepts

Concept Meaning
Target A named R object produced by a function
Dependency A target that another target uses as input
Invalidation Automatic detection that a target needs rerunning

22.4 Quarto for Literate Analysis Reports

Quarto (the system used for this book) is also ideal for individual analysis reports. Embedding code and narrative together ensures that the reported results are always in sync with the code that produced them.

22.4.1 Key report practices

---
title: "LC-MS Metabolomics Report"
author: "Your Name"
date: today
execute:
  echo: false
  warning: false
---

## Detected Features

```{r}
cat("Features detected:", nrow(feature_matrix), "\n")
cat("Samples:", ncol(feature_matrix), "\n")
```

## Annotated Metabolites (Level 1–2)

```{r}
annotations |>
  filter(msi_level <= 2) |>
  select(name, formula, mz, rt, score) |>
  knitr::kable(digits = 3)
```

22.5 Version Control with Git

Every analysis project should be a Git repository. Minimum practice:

# Initialise
git init
echo ".Rproj.user\n.Rhistory\n*.RData\ndata/raw/" >> .gitignore

# Always commit renv.lock
git add renv.lock _targets.R analysis_report.qmd
git commit -m "Initial reproducible workflow"

22.5.1 What to commit vs exclude

Commit Exclude
renv.lock renv/library/ (large, regenerable)
_targets.R _targets/ store (large, regenerable)
.qmd files Raw .mzML files (too large)
references.bib Intermediate result RDS objects

22.6 Data Deposition and FAIR Principles

Published MS data should follow FAIR (Findable, Accessible, Interoperable, Reusable) principles.

22.6.1 Recommended repositories

Repository Data type URL
ProteomeXchange / PRIDE Proteomics raw data + identifications proteomexchange.org
MetaboLights Metabolomics raw data + metadata ebi.ac.uk/metabolights
MassIVE MS data (any type) massive.ucsd.edu
Zenodo Code + small processed data zenodo.org

22.6.2 Minimum metadata to deposit

  • Raw data files (mzML preferred)
  • Sample metadata table (ISA-Tab format for MetaboLights)
  • Analysis scripts or _targets.R
  • renv.lock
  • Processed feature tables
